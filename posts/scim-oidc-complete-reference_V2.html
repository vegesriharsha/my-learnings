2. The protocol family

SSO is the goal. OAuth 2.0 is the authorization foundation. OIDC adds identity on top. SAML is the parallel XML-based universe. See v3 for the full protocol hierarchy diagram (Figure 1) and detailed explanations of each layer.

Key mental model: OIDC is a usage mode of OAuth 2.0, activated by openid in the scope. Same pipes, same endpoints — the scope is the switch. SAML achieves the same goal independently via XML.

3. How OIDC extends OAuth 2.0 — Entra ID as unified provider

Entra ID is simultaneously OAuth 2.0 Authorization Server and OIDC Provider through the same endpoints. With scope=openid email profile: returns ID Token + Access Token (OIDC). Without openid: returns Access Token only (pure OAuth 2.0). See v3 Figure 2 for the scope fork diagram.

4. OIDC vs SAML — two paths to the same destination

See v3 for the full comparison table. Key differences: OIDC uses JSON/JWT with a server-to-server back-channel; SAML uses XML with everything through the browser. OIDC's back-channel is a security advantage — client_secret never touches the browser.

5. OAuth 2.0 four roles — mapped to this architecture

Resource Owner = user. Client = GitHub EMU. Authorization Server = Entra ID. Resource Server = GitHub API (during dev use, not login). See v3 Figure 3.

6. Architecture overview — two protocols, three actors

OIDC handles authentication (who?), SCIM handles provisioning (should they exist?). OIDC flows through the browser; SCIM is server-to-server. See v3 Figure 4 for the full architecture diagram.

7. OIDC authentication flow (detailed)

GitHub EMU (OAuth 2.0 Client) orchestrates the Authorization Code Flow with PKCE (see section 8).

  1. User → GitHub User navigates to GitHub enterprise URL.
  2. GitHub → User HTTP 302 redirect to Entra ID /authorize. Includes client_id, redirect_uri, scope=openid email profile, state, nonce, code_challenge (PKCE).
  3. User → Entra ID Browser follows redirect. Conditional Access policies evaluated (section 10). User authenticates (password+MFA or FIDO2 — section 9). Secrets go to IdP only.
  4. Entra ID → User Issues short-lived authorization code. Redirects to GitHub callback.
  5. User → GitHub Browser delivers auth code. GitHub verifies state.
  6. GitHub → Entra ID Back-channel: POST to /token with code + client_id + client_secret + code_verifier (PKCE). Never touches browser.
  7. Entra ID → GitHub Verifies code_verifier against original code_challenge. Returns ID Token (JWT) + Access Token + Refresh Token (section 11). GitHub validates ID Token via JWKS.
  8. GitHub → User Maps ID Token claims to SCIM-provisioned EMU account. Session cookie set (section 11 covers lifetime).
Prerequisite: OIDC login fails if the user hasn't been SCIM-provisioned first. SCIM creates; OIDC authenticates.

8. PKCE — protecting the authorization code

In section 7, steps 2 and 6 now include code_challenge and code_verifier. This is PKCE (Proof Key for Code Exchange), a security extension that's now considered mandatory for all Authorization Code flows. It solves a specific attack: what if someone intercepts the authorization code in transit between steps 4 and 5?

Figure 5 — PKCE flow: how code_challenge and code_verifier protect the auth code
Why this matters: Without PKCE, a stolen authorization code can be exchanged for tokens. With PKCE, the code is useless without the code_verifier that only the legitimate Client holds. The verifier never travels through the browser — it goes directly from GitHub's server to Entra ID's /token endpoint on the back-channel. PKCE is now required by Entra ID for all new app registrations.

9. Authentication methods at the IdP — what happens inside step 3

OIDC doesn't care how the user authenticates — the IdP handles it internally. GitHub receives the same ID Token regardless. Three methods:

Password + MFA

Type password, then approve MFA push or enter OTP. Two separate steps, two factors (know + have/are).

Phishable — password can be stolen

FIDO2 / WebAuthn

Tap security key or use fingerprint/Face ID. Single gesture satisfies both factors (have + are). FIDO2 = WebAuthn (browser API) + CTAP2 (device protocol).

Phishing-resistant — origin-bound

Passkeys

Same as FIDO2, but credentials sync across devices via cloud (iCloud Keychain, Google, Microsoft). A passkey is a WebAuthn credential.

Phishing-resistant + recoverable

FIDO2 is phishing-resistant because the authenticator cryptographically binds to the exact origin (domain). On evil-login.com instead of login.microsoftonline.com? The key won't respond. The private key never leaves the device. These methods operate underneath OIDC, inside step 3 — GitHub never knows which one was used.

10. Conditional Access — the policy gate between AuthN and token issuance

This is a critical layer we haven't visualized yet. Conditional Access sits inside Entra ID, after step 3 (authentication succeeds) but before the authorization code is issued (step 4). It evaluates context and can block, challenge, or restrict the login.

Figure 6 — Conditional Access sits between authentication and token issuance
Why this matters for SOX: Conditional Access is where you enforce access controls that auditors care about — device compliance, location restrictions, risk-based step-up. These policies are transparent to GitHub (it just sees the login succeed or fail) but they're the enforcement layer that makes your identity perimeter auditable.

11. Token lifecycle, sessions, and the revocation gap

We covered how tokens are issued but not what happens after. This matters enormously for the "employee termination" scenario.

Figure 7 — Token lifetimes and the revocation gap

Token types and lifetimes

  • ID Token (~1 hour): Used once at login to establish identity. Not used after.
  • Access Token (~1 hour): Authorizes API calls. Short-lived by design.
  • Refresh Token (days-months): Silently obtains new Access Tokens without re-authentication. Long-lived.
  • Session cookie (app-specific): GitHub maintains its own session, independent of IdP token expiry.

The revocation gap (SOX concern)

When a user is terminated, SCIM deprovisions their account (next sync cycle, up to ~40 min). But their active GitHub session may persist until the session cookie expires. During this window, the user could still access resources.

Mitigation: CAE (Continuous Access Evaluation) — Entra ID can push critical events (account disabled, password changed, high risk detected) to supported apps, causing near-real-time session revocation. Also: trigger on-demand SCIM sync for terminations.

12. SCIM provisioning flow (detailed)

Push-based, server-to-server. Triggered when a user/group is assigned to the GitHub enterprise app in Entra ID. Direction always: Entra ID → GitHub.

User operations

  • POST /Users — create
  • GET /Users — list/search
  • PATCH /Users/{id} — update
  • PUT /Users/{id} — replace
  • DELETE /Users/{id} — deprovision

Group ops (team mapping)

  • POST /Groups
  • PATCH /Groups/{id}
  • DELETE /Groups/{id}

Base: api.github.com/scim/v2/enterprises/{enterprise}/

Sync cycles

  • Initial: full enumeration via POST
  • Incremental: ~40 min (PATCH/DELETE)
  • On-demand: manual from Entra portal

Attribute mapping

  • userPrincipalName → userName
  • displayName → displayName
  • mail → emails[work].value
  • objectId → externalId

SCIM auth

PAT bearer token — completely unrelated to OIDC credentials.

13. Identity lifecycle — how SCIM and OIDC connect

Assign in Entra → SCIM provisions → user logs in via OIDC → SCIM syncs ongoing → SCIM deprovisions. SCIM must complete before OIDC login is possible. See v3 Figure 6 for the timeline visualization.

Why SCIM instead of JIT?

JIT only creates accounts at first login. No ongoing sync, no group management, no deprovisioning. GitHub EMU requires SCIM — JIT is not supported. SCIM is the enterprise approach: full lifecycle, server-to-server, pre-staged before first login.

14. OAuth 2.0 grant types — beyond the Authorization Code flow

The Authorization Code flow (section 7) is for user-in-the-browser scenarios. OAuth 2.0 defines other grant types for different contexts:

15. OIDC Discovery and logout

The Discovery document — bootstrap for the entire trust relationship

We mentioned .well-known/openid-configuration as a URL, but it's worth understanding what this single JSON document contains. It's how GitHub discovers everything it needs about Entra ID automatically:

GET https://login.microsoftonline.com/{tenant-id}/v2.0/.well-known/openid-configuration

Returns: authorization_endpoint (where to redirect), token_endpoint (where to exchange code), jwks_uri (where to get signing keys), issuer (expected iss claim), scopes_supported, response_types_supported, grant_types_supported, end_session_endpoint (logout)

If this URL becomes unreachable or returns wrong data, the entire OIDC trust chain breaks — GitHub can't find any endpoint. It's a single point of failure and should be monitored.

OIDC Single Logout (SLO) — fragile in practice

We covered logging in but not logging out. If a user logs out of GitHub, should they be logged out of Entra ID and all other apps? That's Single Logout. OIDC supports two mechanisms:

Front-channel logout

GitHub redirects the browser to Entra ID's end_session_endpoint. Entra ID then loads hidden iframes pointing to every app's logout URL. Relies on browser behavior — ad blockers, cookie policies, and network issues can silently break it.

Back-channel logout

Entra ID sends a server-to-server POST with a logout token to each app's registered back-channel logout URI. More reliable than front-channel, but the app must implement the endpoint and process the token.

Practical reality: Full SLO is fragile and many implementations don't fully support it. Logging out of one app often does not log you out everywhere. For the employee termination scenario, rely on SCIM deprovisioning + CAE (section 11) rather than SLO.

16. GitHub EMU wears three hats

Hat 1: OAuth 2.0 Client OIDC login

Initiates auth, redirects to Entra ID, receives code, exchanges for tokens via back-channel (with PKCE), validates ID Token, maps claims to EMU account, creates session. Active orchestration.

Hat 2: SCIM server provisioning

Roles flip. GitHub hosts SCIM API endpoints; Entra ID calls in. Authenticated via PAT bearer token. GitHub is the server.

Hat 3: Resource Server developer use

When devs run git push or call the API (sometimes via Device Code flow), GitHub protects its resources and validates credentials.

18. Trust relationships

See v3 Figure 7 for the full trust chain diagram. Summary: GitHub trusts Entra ID via JWKS signature verification (OIDC). Entra ID trusts GitHub via pre-configured SCIM endpoint + PAT. User credentials (password/MFA/FIDO2) go only to Entra ID, never to GitHub. Two secrets in play: client_secret (OIDC back-channel) and PAT bearer token (SCIM). Two public mechanisms: JWKS keys and OIDC discovery document.
