2. Architecture overview — two protocols, three actors

SCIM and OIDC serve fundamentally different purposes but work together. OIDC handles authentication (who is this user?), while SCIM handles provisioning (should this user exist in GitHub, and with what attributes?). The critical distinction: OIDC flows through the user's browser, SCIM is a direct server-to-server channel.

Key distinction: OIDC requires the user's browser as an intermediary (redirect-based). SCIM is a direct server-to-server REST API call — the Entra ID provisioning service pushes identity data to GitHub's SCIM endpoint without any user involvement.

3. OIDC authentication flow (detailed)

OpenID Connect uses the Authorization Code Flow to authenticate users. This is a multi-step redirect dance between three parties. Every arrow below shows exactly who talks to whom and in which direction.

Figure 2 — OIDC Authorization Code Flow sequence
  1. User → GitHub User navigates to the GitHub enterprise URL in their browser.
  2. GitHub → User GitHub detects no active session and returns an HTTP 302 redirect to Entra ID's /authorize endpoint. The redirect URL includes: client_id, redirect_uri, scope=openid email profile, a random state parameter (CSRF protection), and a nonce (replay protection).
  3. User → Entra ID The browser follows the redirect. Entra ID presents the login page. The user authenticates with credentials and MFA if configured. This is the only step where the user provides secrets — and they go to the IdP, never to GitHub.
  4. Entra ID → User After successful authentication, Entra ID issues a short-lived, single-use authorization code and redirects the browser back to GitHub's callback URL with the code and original state parameter.
  5. User → GitHub The browser delivers the authorization code to GitHub's callback endpoint. GitHub verifies the state parameter matches what it originally sent.
  6. GitHub → Entra ID Back-channel (server-to-server): GitHub's server directly calls Entra ID's /token endpoint, sending the authorization code + client_id + client_secret. This never passes through the browser.
  7. Entra ID → GitHub Entra ID validates the code and returns an ID Token (JWT containing user identity claims: sub, email, name, groups) and an Access Token. GitHub validates the ID Token's signature by fetching Entra ID's public keys from the JWKS (JSON Web Key Set) endpoint.
  8. GitHub → User GitHub maps the ID Token claims to the corresponding EMU user account (which must have been pre-provisioned via SCIM). A session cookie is set, and the user is logged in.
Prerequisite: OIDC authentication will fail if the user has not been provisioned via SCIM first. The EMU account must exist in GitHub before the user can authenticate. SCIM creates the account; OIDC authenticates it.

4. SCIM provisioning flow (detailed)

SCIM is a push-based, server-to-server protocol. Entra ID's provisioning service calls GitHub's SCIM REST API endpoints to manage the full lifecycle of user and group identities. No user interaction is required.

Figure 3 — SCIM provisioning lifecycle

SCIM provisioning cycle

  • Initial sync: Entra ID enumerates all assigned users/groups and creates them in GitHub via POST
  • Incremental sync: Runs every ~40 minutes by default; detects changes in Entra ID and pushes PATCH/DELETE
  • On-demand: Manual sync triggered from Entra ID portal

Attribute mapping examples

  • userPrincipalName → userName
  • displayName → displayName
  • mail → emails[type eq "work"].value
  • givenName → name.givenName
  • surname → name.familyName
  • objectId → externalId
Key point — SCIM authentication: The SCIM channel authenticates via a PAT (Personal Access Token) generated by a GitHub enterprise owner. This PAT is stored as a secret in the Entra ID enterprise application's provisioning configuration. It is not related to OIDC tokens — SCIM and OIDC use entirely separate authentication mechanisms.

8. Failure modes — what breaks and why

SymptomProtocolRoot cause → fix
User can't log in — "account not found" SCIM User hasn't been provisioned yet. Check: Is the user assigned to the GitHub enterprise app in Entra ID? Has the provisioning cycle run? Check Entra ID provisioning logs.
Login redirect fails — "redirect_uri_mismatch" OIDC The callback URL configured in GitHub doesn't exactly match the redirect URI registered in the Entra ID app registration. URLs must be identical including trailing slashes.
Token validation fails — "invalid signature" OIDC GitHub can't verify the ID Token. Either the JWKS endpoint is unreachable (network/firewall), keys have rotated and GitHub's cache is stale, or the issuer URL is wrong.
SCIM provisioning errors — 401 Unauthorized SCIM The PAT (Personal Access Token) stored in Entra ID has expired or been revoked. Generate a new PAT in GitHub and update the Entra ID provisioning config.
User attributes out of sync SCIM Attribute mappings in Entra ID are incorrect, or the incremental sync hasn't run. Verify mappings and trigger a manual provisioning cycle.
User deprovisioned unexpectedly SCIM User was unassigned from the GitHub enterprise app in Entra ID, or their Entra ID account was disabled/deleted. SCIM propagated the change automatically.
